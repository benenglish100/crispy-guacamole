Field 3: the driver — the discrepancy ★ The reveal
Avatar says
“Here's the gap. On the call, the customer asked to add a named driver — but on the policy, they're not there. That's exactly the disconnect: a good call, but the system doesn't match it. Flagging this one.”
Ben builds

Plant the discrepancy here. Pre-stage the W policy so the agreed-to-add driver is not on it. additional_driver is a high-weight name-present check (BuildSpec §4) — so a single FLAG legitimately trips the review trigger in Scene 9.

Data

Transcript: customer asks to add a driver. W policy: driver absent. The single deliberate mismatch.

Why an added driver, not a removed one: the agreed-removal field (cover_removed) is medium weight in the build spec, so a flag there wouldn't on its own justify “high-weight field failed → review.” Planting on additional_driver (high) keeps Scene 9's trigger logic honest — and “was the driver added” is one of Daniel's literal examples. If Admiral prefers a removal, raise cover_removed to high first and confirm, or trip via the score-below-threshold path.

What Ben needs from Admiral first

Gather before the build — item 1 is make-or-break.

  1. A representative, sanitised policy-management screen flow (CRM / customer-care portal, “W” instance). Without it, screen-understanding reliability (⚠️C) is untested. The make-or-break input.
  2. One anonymised real motor call transcript — quoted price + added product + agreed driver add (ask Daniel — Admiral wants a real call).
  3. A matching W-policy record, pre-staged with the driver-add discrepancy planted (driver absent).
  4. Admiral's current QA / accuracy pass bar — to tune the placeholder <90% threshold to their real standard.
  5. Cornerstone cloud-vs-legacy answer (open since 29 May) — before any xAPI write-back is promised.
  6. Data-residency answer — where screen content + transcript get processed/stored, for Admiral's UK FSI security review.
